# L1.1 — Agent Memory as Repo Files (design)

**Date:** 2026-07-09
**Status:** Approved design, **blocked on two probes** (§2.2, §8.1) before implementation.
Revised after review: the permission hedge in §2 resolved against the design (§2.1), and
`supersedes`, the mcp-config shape, the slug render, and frontmatter serialization all
needed correcting.
**Supersedes:** the central decision of `2026-07-09-l1-memory-substrate-design.md`
("agent memory is gortex memory; skep stores nothing"). That spec's L1 code is
merged; this revision replaces its *substrate* while keeping most of its plumbing.

---

## 1. Decision

**Agent memory is tracked files in the repo. skep owns the write path.**

L1 (merged this morning) delegated memory to the machine-wide `gortex memory` CLI.
It worked, but the store is daemon-owned and opaque: you cannot open it, diff it,
grep it, review what an agent claimed to learn, or carry it to another machine.
Memory you cannot inspect is memory you cannot trust.

So memory becomes what it should have been: **one Markdown file per fact, tracked in
git, under `<repo>/.agent-memory/`.** Git gives inspection, diffing, review, revert,
and cross-machine sync for free. One file per fact means concurrent agents never
produce a merge conflict.

skep's contribution is two halves of an asymmetry:

- **Reading is free.** At spawn, skep reads `<repo>/.agent-memory/` and folds the
  facts directly into the `--append-system-prompt` addendum it already sends. No tool
  call, no permission grant, no daemon, no `--add-dir`. The agent *starts* knowing.
- **Writing needs a channel.** A `MemoryShim` MCP server — stdio, alongside the
  `MailboxShim` skep already ships (§5.2) — exposes one tool, `remember`. skep writes the
  file. The agent never touches its `Write` tool.

The gortex dependency is deleted. `gortex` remains valuable and remains installed; it
simply indexes `.agent-memory/*.md` as ordinary repo source, so its search still works
over these facts — for free, and without skep depending on a daemon.

Per-task scratch stays a plain file in the agent's worktree. No mechanism needed.

## 2. What was verified (2026-07-09, `claude` CLI on this host)

Recorded because four plausible-sounding assumptions turned out false, and one of them
had already been shipped.

| Claim | Verdict |
|---|---|
| An untracked file in `<repo>/.claude/` reaches the agent | **False.** Agents run in `worktrees_root/<repo>-<tid>`; `git worktree add` produces a clean checkout. Untracked files do not follow. Tracked files do. |
| `.claude/**` is writable by an agent | **False.** It is a harness-protected sensitive path. Blocked even under `--permission-mode acceptEdits`. **Config-independent** — this killed the originally-chosen location. |
| A headless `-p` agent can write files by default | **False.** Every write is denied, inside cwd or out. `--permission-mode acceptEdits` unlocks writes inside cwd; `acceptEdits` + `--add-dir` unlocks writes outside it. |
| A headless `-p` agent can call an MCP tool by default | **False.** The call is denied. **`--allowedTools mcp__<server>__<tool>` unlocks it.** `acceptEdits` does **not** — edits and MCP tools are separate permission axes. |
| L1's claim that `--append-system-prompt` and `--add-dir` exist | **True.** Both present in `claude --help`. |

### 2.1 As configured on this host, skep agents run with no permissions at all

An earlier revision of this section hedged: *"unverified, and this spec does not depend
on it — whether the worker profile (`SKEP_CLAUDE_CONFIG_DIR`) already grants these
permissions."* It does not. Checked on this host, 2026-07-09:

- `SKEP_CLAUDE_CONFIG_DIR` is unset — no shell env, no systemd unit, no env file.
- So `WorkerConfig.claude_config_dir` is `None`, `_agent_env` never sets
  `CLAUDE_CONFIG_DIR`, and agents inherit the default `~/.claude` profile.
- `~/.claude/settings.json` has **no `permissions` block**, and there is no config-root
  `settings.local.json` (which the harness would not read anyway).
- `AgentProcess._argv` passes no `--permission-mode` and no `--allowedTools`.

Applying the table above to that configuration yields two consequences, not one:

1. **The shipped `MailboxShim` is unreachable.** `send_message` / `read_inbox` are MCP
   tools with no grant; the call is denied. Its only end-to-end test is skipped unless
   `SKEP_RUN_INTEGRATION=1`, which is why nothing caught it.
2. **Agents cannot write or edit files either.** Every `Write` and `Edit` is denied.
   (Whether read-only tools still pass was not probed — §2's table established the write
   and MCP denials only. Assume nothing about what an agent *can* do here.) So skep has
   no permission baseline as configured on this host. That this went unnoticed is
   consistent with §8.1: agent execution is unmeasured.

The `_argv` facts are config-independent. The profile facts are not: another worker host
could set `SKEP_CLAUDE_CONFIG_DIR` at a profile that grants everything. Nothing in this
spec relies on which is true — skep will pass its grants explicitly — but §2.3's baseline
question must be answered for the hosts agents actually run on, not just this one.

**This widens the spec's scope, and the widening is not optional.** §5's grant cannot be
designed as if it sits on top of a working agent, because there isn't one.

### 2.2 The one thing still unverified — and the write path dies if it goes the wrong way

**Does CLI `--allowedTools` *merge with* the profile's allowlist, or *replace* it?**

§2 verified that `--allowedTools mcp__<server>__<tool>` unlocks that tool. It did **not**
verify that adding the flag leaves every other permission intact. If the flag replaces
rather than merges, then `--allowedTools mcp__memory__remember` becomes the agent's
entire permission set and strips whatever `Bash` / `Edit` grant §2.1 establishes it needs.

This cannot be settled by reading config. It needs one real spawn: an agent launched with
the flag, confirmed to still run `Bash` **and** call `remember`. **That probe blocks §5.**

### 2.3 Consequence for §5's framing

§5 below argues the narrow grant is "far tighter than `--permission-mode acceptEdits`,
which would hand the agent blanket write access." Given §2.1 that comparison is between
two things that are not substitutes: an agent doing coding work needs write access to its
worktree regardless of memory. The memory grant is **additive to a permission baseline
skep has not yet written**, not an alternative to it. Deciding that baseline is the same
one-spawn experiment as §2.2, and this spec no longer treats it as out of scope.

## 3. The store

```
<repo>/.agent-memory/
    stack-takes-90s-to-come-up.md
    ws-reconnect-needs-jitter.md
    integration-tests-flake-under-load.md
```

Tracked in git. One fact per file. Written into the **parent repo** (`repos_root/<repo>`),
never the agent's worktree — so a fact survives task failure and branch abandonment,
which is precisely when the best facts are learned. Files land untracked in the parent's
working tree; **the owner commits them.** That review step is the accountability the
design is buying.

File format:

```markdown
---
title: Stack takes 90s to come up
kind: gotcha
created: 2026-07-09T14:22:03Z
superseded_by: null
---

`docker compose up` returns before the API is healthy. Poll /healthz, do not sleep.
```

`kind` is one of `gotcha`, `constraint`, `decision`, `convention`, `incident` — the
vocabulary agents already know from other memory tooling. `superseded_by` holds a slug
or `null`.

### 3.1 The frontmatter is hand-parsed, and `title` is agent-supplied

skep has no YAML dependency (`aiogram`, `aiohttp`, `zeroconf`, `mcp`), and this spec does
not add one for four keys. So the frontmatter is parsed by hand — and `title` is free text
chosen by a language model. A title containing `:` or a newline breaks a naive
`key, _, value = line.partition(":")` reader.

That failure is not benign. §6 says malformed frontmatter means *skip that file*, so a
badly-titled memory **silently disappears from the addendum on the next read** — the agent
writes a fact, is told the write succeeded, and the fact is never seen again.

skep writes these files, so skep controls serialization. Close it at the write end:

- The **slug is the identity**; `title` is presentation only.
- Serialize `title` as a quoted scalar with `"` and `\` escaped, and reject or strip
  newlines. The reader unquotes symmetrically.
- `kind` is validated against the closed vocabulary above; `created` is ISO-8601 emitted
  by skep, never by the agent; `superseded_by` is a slug or `null`.

Every frontmatter value is therefore machine-generated or escaped by skep. The
skip-on-malformed rule in §6 stays as a guard against hand-edits and merge artifacts —
not as the routine outcome of an agent using a colon.

**Consequence of "tracked":** the parent repo's working tree goes dirty as agents learn.
That is intended and visible, not a side effect to hide.

## 4. Reading — free, at spawn

`MemoryStore` implements L1's existing `MemoryProbe` protocol — `async addendum_for(repo_path)
-> str | None` — so `Supervisor` needs no change to how it asks for an addendum. Internally:
`_load()` reads `.agent-memory/*.md`, skips files whose `superseded_by` is set, sorts
newest-first by `created`; `_render()` folds them into the addendum text below. It returns
`None` only when memory is unavailable (§6), never merely because the store is empty.

```
## Memory

Durable facts other agents learned about this repo:

- [stack-takes-90s-to-come-up] **Stack takes 90s to come up** (gotcha): `docker
  compose up` returns before the API is healthy. Poll /healthz, do not sleep.
- ...

Record a memory when you learn something that would save the next agent time and
that the repo itself does not already record — an operational fact, a constraint you
discovered the hard way, or a decision and its reasoning:
    the `remember` tool (title, body, kind, supersedes)

If a fact above is now wrong, supersede it rather than adding a contradiction. Pass
the slug in brackets, exactly as shown above:
    remember(title=..., body=..., supersedes="stack-takes-90s-to-come-up")

Do NOT record what the repo already records — code structure, git history, CLAUDE.md.
Scratch notes for this task alone: write a file in your worktree.
```

**The bracketed slug is load-bearing, not decoration.** An earlier draft rendered
`- **Title** (kind): body` and then asked the agent for `supersedes="<slug>"`. Slugs
appeared nowhere in the addendum, and §5.1's `-2` collision suffix makes them unguessable
from a title. Supersession was unusable as specified: the agent could only guess a
filename it had never been shown. The renderer emits each fact's slug verbatim.

Phrased prescriptively — when to write, not merely how. Prescriptive trigger conditions
measurably lift correct tool use (carried forward from L1 §3, which carried it from the
`claude-api` skill's tool-description guidance).

**Bounded.** At most `SKEP_MEMORY_MAX_BYTES` (default 8192) of rendered facts, newest
first. When facts are dropped, the addendum says so explicitly (`… N older memories
omitted`) rather than silently truncating — an agent that believes it has seen all the
memory is worse off than one that knows it hasn't.

**Zero memories** (the common case for a fresh repo) omits the recall section but keeps
the write instructions: an agent with nothing to recall can still learn something.

## 5. Writing — one MCP tool

`MemoryShim` serves exactly one tool:

```python
remember(title: str, body: str, kind: str = "gotcha", supersedes: str | None = None) -> str
```

It returns the written path, so a failed write reaches the agent as a tool error rather
than a silent no-op it believes succeeded.

Spawn passes `--allowedTools mcp__memory__remember`, plus the mailbox's tools when the
mailbox is active, plus whatever baseline §2.3 settles on. That grant is a *superset* of
what memory needs, and per §2.2 it must be confirmed by a real spawn that the flag
**merges with** rather than replaces the agent's other permissions before any of this is
built.

### 5.1 Path safety is the whole security story

`title` becomes a **filename**, and the caller is a language model. A title of
`../../../.ssh/authorized_keys` must not escape the memory directory.

1. Slugify to `[a-z0-9-]+` (lowercase, collapse runs, strip leading/trailing `-`).
2. Reject an empty slug.
3. Build the path, `Path.resolve()` it, and assert it is under the resolved
   `<repo>/.agent-memory/`. Reject otherwise.

Step 3 is not redundant with step 1. It is the assertion that holds even if the slugifier
is later "improved" to allow a character it should not. It gets its own tests.

**`supersedes` is the second path, and it takes the same three steps.** An earlier draft
declared path safety "the whole security story" and then validated only `title`.
`supersedes=<slug>` names a file that skep **opens and mutates**; `supersedes=
"../../../.ssh/authorized_keys"` escapes through exactly the door this section exists to
close. Both arguments run steps 1–3, and `supersedes` additionally must resolve to a file
that already exists inside the memory directory — creating one by superseding it is not a
thing.

**Collisions.** A *new* fact whose slug already exists gets a `-2` suffix rather than
overwriting another agent's memory. A `supersedes=<slug>` call sets `superseded_by` in
the named file and writes the new one — the old file stays on disk, auditable, and stops
being injected.

### 5.2 Transport: stdio, not HTTP

`MailboxShim` is an in-process HTTP server for two reasons that do not apply here: it
bridges to the queen across the network, and it closes over a per-agent `tid` in the
supervisor to make `from` spoof-proof. `remember` needs neither — it is a **stateless
local file write** parameterized by one repo path. Mirroring MailboxShim's transport
would inherit a per-spawn uvicorn server, an ephemeral port, and a bearer token passed on
the agent's command line: machinery for a hop that never happens.

`MemoryShim` is therefore a **stdio** MCP server — `--mcp-config` names a subprocess
command (`{"type": "stdio", "command": ..., "args": [<parent repo path>]}`) instead of a
URL. No port, no token, no listening socket, and it sidesteps the same-UID
argv-disclosure caveat documented at `worker/mcp_shim.py:32`. A stdio entry and the
mailbox's `{"type": "http"}` entry coexist in the one `mcpServers` map §7 describes.

**The lifecycle differs from `MailboxShim`, and §7's inventory must reflect it.** The
mailbox shim is constructed, `start()`ed, and `stop()`ed by `Supervisor`, which tracks it
in `self._shims` and tears it down in `run_events`' `finally`. A stdio server has none of
that: `claude` spawns it as its own child and reaps it on exit. `Supervisor` contributes
a dict entry and nothing else — no `_shims` bookkeeping, no teardown path, no leak on a
failed spawn. The parent repo path reaches the tool as a command-line argument, so the
agent cannot influence which repo it writes to, the same closed-over-identity property
`tid` gives the mailbox.

This is a decision made during review, not one inherited from the original draft — worth
flagging because it is the one place this revision *adds* to the design rather than
correcting it.

## 6. Graceful degradation — unchanged invariant, easier to keep

**Nothing about memory may fail a spawn.** L1's soft-dependency rule survives; only its
failure modes change. On the **read** path there is no daemon, no subprocess, no timeout —
just filesystem reality. (The write path's stdio server is a subprocess, but it is
`claude`'s child, spawned after skep's `spawn()` has returned; it cannot fail a spawn
either. §5.2.)

`MemoryStore._load()` returns `[]` and logs once, never raising, when:

- `.agent-memory/` does not exist (the common case — no memory yet)
- it is unreadable
- a file has malformed frontmatter — **skip that file, keep the rest.** One bad file must
  not blind the agent to the other twenty.

`SKEP_MEMORY_ENABLED=false` (default true) omits everything: no addendum section, no
shim, no `--allowedTools` grant.

**Read failure does not disable writing.** `SKEP_MEMORY_ENABLED` is the only switch that
takes down the write channel. An unreadable or absent `.agent-memory/` yields no recall
section, but the shim still starts and the grant is still passed — an agent with nothing
to recall is exactly the agent most likely to learn something worth recording, and the
first `remember` call creates the directory.

Config surface, both on `WorkerConfig`: `memory_enabled` (`SKEP_MEMORY_ENABLED`, default
true — already exists) and `memory_max_bytes` (`SKEP_MEMORY_MAX_BYTES`, default 8192 — new,
the §4 cap).

## 7. What is reused, and what is deleted

L1's plumbing was built as dumb passthrough and does not care what the text says.

**Survives untouched:** `AgentProcess.append_system_prompt` → `--append-system-prompt`;
`Supervisor`'s `memory` injection point and its soft-fail guard; `WorkerConfig.memory_enabled`
/ `SKEP_MEMORY_ENABLED`; the conditional-kwarg pattern.

**Deleted:** `recall_command`, `probe_memory`, `PROBE_TIMEOUT`, and the entire gortex
daemon dependency — along with the preflight probe's reason for existing.

**Transformed:** `MemoryPreflight` → `MemoryStore` (read the dir, render the addendum).
The `MemoryProbe` protocol keeps its exact shape — `async addendum_for(repo_path) -> str | None`
— so `Supervisor` and its tests are untouched; only the implementation behind it changes.

**Added:** `MemoryShim` — a stdio MCP entry point, **not** a `MailboxShim`-style
supervisor-owned server (§5.2): no `start()`/`stop()`, no `_shims` entry, no teardown in
`run_events`. Also `AgentProcess.allowed_tools` → `--allowedTools`, and the
`.agent-memory/` store itself.

**Restructured — and an earlier draft of this inventory omitted it.** `AgentProcess`
currently carries a *single* optional server: `mcp_url` / `mcp_token` fields that
`_argv` (`agent.py:79-84`) hardcodes into `{"mcpServers": {"mailbox": server}}`. Two
servers do not fit that shape, and memory must work when the mailbox is off (no
`mailbox_client`, so no `mcp_url` today). `AgentProcess` therefore takes an **N-server
map** — `mcp_servers: dict[str, dict]`, rendered whole into `--mcp-config` — and
`Supervisor` contributes `mailbox` and `memory` entries independently. That is a
signature change on both classes and on `test_agent_mcp.py`, not the "dumb passthrough
survives untouched" this section otherwise describes. The `memory` key is what makes the
grant name `mcp__memory__remember` correct.

Nothing merged this morning is wasted except the gortex-specific functions, which were
always the part the spec called replaceable.

## 8. Testing

Everything is filesystem and string work — all unit-testable with `tmp_path`, no daemon
to fake. Strictly better than what L1 could test.

**Read path:** memories render into the addendum; a missing directory yields no recall
section but keeps the write instructions; one malformed file does not suppress the
others; the byte cap truncates newest-first and says how many were omitted; superseded
files are excluded from injection but remain on disk; a spawn succeeds when
`.agent-memory/` is unreadable.

**Write path (the tests that matter most).** Every case below runs twice — once against
`title`, once against `supersedes` (§5.1): `../escape` is rejected; an absolute path is
rejected; input that slugifies to empty is rejected; a resolved path outside the memory
dir is rejected even if the slug looks clean. Then: a `title` collision produces `-2`
rather than an overwrite; `supersedes` naming a file that does not exist is rejected
rather than creating one; `supersedes` marks the old file and leaves it on disk.

**Serialization (§3.1):** a title containing `:` round-trips through write→read; a title
containing a newline is rejected or stripped at write; `kind` outside the closed
vocabulary is rejected; a fact written by `remember` is always readable by `_load()`
(the property that §6's skip-on-malformed rule would otherwise hide).

**Render (§4):** each fact's slug appears in the addendum, including a `-2` collision
slug — the string the agent must pass back as `supersedes`.

**Spawn wiring:** `--mcp-config` carries a `memory` server when the mailbox is off, and
both `memory` and `mailbox` when it is on; `--allowedTools` is present when memory is on
and absent when `SKEP_MEMORY_ENABLED=false`; an unreadable `.agent-memory/` still yields
a shim and a grant (§6); the addendum names the parent repo's memories, not the
worktree's.

### 8.1 What tests cannot show — and what must be probed before writing code

Two of these now **block implementation** rather than merely trailing it.

- **Blocking (§2.2):** that `--allowedTools` merges with, rather than replaces, the
  agent's other permissions. One real spawn: pass `--allowedTools mcp__memory__remember`,
  confirm the agent can still `Bash` *and* call `remember`. If it replaces, §5's grant
  must enumerate every tool the agent needs, and the design of §2.3's baseline changes.
- **Blocking (§2.1/§2.3):** what permission baseline a skep agent actually needs to do a
  coding task at all, given it has none today. Same spawn answers it.
- That agents actually *use* the tool. Unmeasured. L1 §5 said "measure before assuming it
  works," and that remains true and remains undone.

## 9. Risks

- **Agents may ignore the instruction.** Unfixable in code; mitigated by prescriptive
  phrasing. Measure before assuming.
- **Agents may write low-value or duplicate memories.** The owner reviews before
  committing — the store cannot silently rot. Consolidation is deferred (§10).
- **`title` and `supersedes` are both attacker-controlled input to a filesystem path.**
  Mitigated by §5.1, which is the highest-value test surface in this spec.
- **skep agents have no permission baseline as configured on this host** (§2.1). Memory's
  write channel cannot be designed, and this spec cannot be implemented, without settling
  what that baseline should be. Its resolution is now in scope.
- **`--allowedTools` may replace rather than merge** (§2.2). If so, shipping §5 as
  written would strip agents of the tools they need to work. Probe first.
- **The parent working tree goes dirty during a run.** Intended and visible. An owner who
  never commits accumulates untracked files.
- **Concurrent `remember` calls to the same slug** race on the `-2` suffix. One file per
  fact keeps the blast radius to one memory; a lost duplicate is not a correctness bug.
- **Two agents on two hosts** produce divergent `.agent-memory/` sets until both branches
  merge. That is git working as intended, and it is strictly better than L1, where the two
  hosts' memories never met at all.

---

## 10. Deferred, not discarded

- **The queen-hosted store** (L1 §7). Still right for L2's persistent managers; still not
  needed until agents on different hosts must share memory without merging git.
- **Consolidation / "sleep cycle."** Presupposes a populated store. Its own spec, once
  memory is accumulating and there is evidence it bloats.
- **Vectors.** No first-party embeddings endpoint; gortex indexes these files anyway.
- **Repairing the skipped mailbox e2e test.** Confirmed bug, not a suspected one (§2.1):
  the mailbox is unreachable today because no grant exists. Its own spec — but it is
  fixed *incidentally* here, since §2.3's baseline and §5's `--allowedTools` plumbing are
  what the mailbox was missing. The follow-up is the test, not the fix.
